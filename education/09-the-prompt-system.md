# 09 — The prompt system

## The problem

Every request to a provider carries a system prompt. If that prompt is assembled
through scattered string concatenation, three things go wrong:

1. **Drift between the prompt and the real tool set.** An engineer adds a tool
   to the registry but forgets to update the prompt prose, or removes one but
   leaves "use the write tool" in the instructions. The model receives
   contradictory advice — it tries to call a tool that does not exist, or skips
   one that does because the prompt never mentioned it.

2. **Silent partial rendering.** When capabilities are boolean flags constructed
   by hand (`hasRead := readTool != nil`), the prompt builder is a second
   registry that must stay in sync. A miss causes a tool-specific instruction
   block to appear or disappear without any compilation error.

3. **Injection and escaping.** User-supplied values — file paths, skill names,
   context file content — get interpolated into an XML-structured prompt. String
   concatenation without consistent escaping lets a path containing `<`, `&`, or
   `"` corrupt the prompt structure.

The harness solves all three with a single pipeline: a typed `PromptData` struct,
a `go:embed`-ed `text/template`, and a `CapabilitiesSection` function that
derives the authoritative tool list from the live registry on every build.

## Key decisions and the thinking process

### Decision 1: a typed struct, not `map[string]any`

The template renderer receives `PromptData` — a concrete struct with fields for
every conditional section. There is no `map[string]any`, no `render(data
map[string]any) string`, and no runtime reflection to check whether a key
exists.

`PromptData` has 26 fields (`internal/prompt/system.go:30-62`). Each `HasXxx`
boolean gates exactly one `{{if .HasXxx}}` block in the template. The struct is
the contract: you cannot add a template conditional without adding the field, and
you cannot forget to compute the field without a compilation error at the call
site.

The alternative — `map[string]any` — was considered and rejected because a
missing key renders silently as the zero value (empty string, `false`). You would
not discover the omission until a human reads the rendered prompt and notices a
missing section. Worse, a typo in a key name compiles fine and renders nothing.

### Decision 2: embedded template with `go:embed`, not string concatenation

The prompt lives as `internal/prompt/templates/system.md.tmpl` — 127 lines of
Markdown with `text/template` directives. It is embedded at compile time with
`//go:embed templates/system.md.tmpl` (`internal/prompt/system.go:64-65`). The
template is parsed once at package initialization (`template.Must`, line 75) and
re-executed for every prompt build.

The alternative — string concatenation spread across `if hasRead { prompt +=
"..." }` blocks — was rejected because it intermixes structure and content. A
template keeps the prompt readable as one document; you can see the full shape
and the conditional gates without tracing through multiple functions.

The template also registers an `xml` function for safe escaping
(`internal/prompt/system.go:75-77`):

```go
var xmlEscaper = strings.NewReplacer(
    "&", "&amp;", "<", "&lt;", ">", "&gt;",
    `"`, "&quot;", "'", "&apos;",
)
```

Every user-supplied value in the template passes through `{{xml .Path}}` or
`{{xml .Name}}`, preventing XML injection from file paths, skill names, or
context file content.

### Decision 3: capability projection from the live registry

The `<available_tools>` block is not hand-written prose. It is generated by
`CapabilitiesSection(names []string, snippets map[string]string) string`
(`internal/prompt/prompt.go:114-141`). It iterates the actual tool names selected
for this session and emits one `- name: description` line per tool. A tool
without a snippet is skipped (a safety net; the production path always provides
snippets via `BuildToolSnippets`).

The invariant: **the prompt never names a tool the registry does not expose.**
This is enforced by construction — the `names` slice comes from
`ToolNames(active)` (`internal/session/session.go:92-98`), which reads
`tool.Name` from each `agent.AgentTool` in the active set. There is no separate
prompt-side tool list to maintain.

Similarly, the `HasRead`, `HasBash`, `HasEdit` booleans are derived from
`hasTool(tools, "read")` etc. (`internal/prompt/prompt.go:70-82`). If a tool is
absent from the registry, its boolean is `false` and the corresponding template
block renders nothing. The test `TestBuildSystemPromptProjectsOnlyActiveCapabilities`
(`internal/prompt/prompt_test.go:44-80`) explicitly asserts that inactive tools
leave no trace in the prompt and that unavailable schemes (`vault://`, `mcp://`,
`xd://`) never appear.

### Decision 4: custom prompt path preserves capabilities

The template wraps its entire body — role, principles, runtime, execution
workflow, delivery contract — in `{{if .HasCustomPrompt}}{{.CustomPrompt}}{{else}}…{{end}}`
(`internal/prompt/templates/system.md.tmpl:1,101`). A custom prompt replaces the
default body but retains:

- The capabilities section (`{{.CapabilitiesSection}}`)
- Appended content (`{{.AppendSystemPrompt}}`)
- Context files (`<project_context>`)
- Skills block (`<available_skills>`)
- Date and working directory footer

This means an integration can supply its own role and instruction text without
losing the mechanically-derived capability list or the context/skill projection.
The test at `internal/prompt/prompt_test.go:83-116` verifies that custom text
appears but the default `<system-conventions>` and `# Engineering Principles`
blocks do not.

### Decision 5: skills gated on the read tool

Skills are only listed in the prompt when the `read` tool is active
(`internal/prompt/prompt.go:64`):

```go
HasSkills: hasTool(tools, "read") && len(promptSkills) > 0,
```

The reasoning: skills are loaded by reading `skill://<name>` through the read
tool. If the session has no read tool, listing skills in the prompt would tell
the model about capabilities it cannot use. The test
`TestBuildSystemPromptSkillsOnlyWithRead`
(`internal/prompt/prompt_test.go:119-141`) confirms that `<available_skills>`
appears with `read` and disappears without it.

## Signatures and types

### PromptData (`internal/prompt/system.go:30-62`)

The typed data model that drives template rendering. Every field corresponds to
exactly one template directive. Boolean `HasXxx` fields gate conditional blocks;
slices drive `{{range}}` loops; the pre-rendered `CapabilitiesSection` string is
injected as a block.

```go
type PromptData struct {
    ProductName         string          // brand-neutral product name ("Harness")
    CustomPrompt        string          // replacement for default template body
    HasCustomPrompt     bool            // true when CustomPrompt is non-empty
    CapabilitiesSection string          // pre-rendered <available_tools> + <runtime> block
    AppendSystemPrompt  string          // appended after the main template body
    Cwd                 string          // current working directory, slash-normalized
    Date                string          // ISO date for the "Current date:" footer
    ContextFiles        []ContextFile   // project context injected under <project_context>
    HasContextFiles     bool            // true when ContextFiles is non-empty
    Tools               []PromptTool    // tool names and labels for the prompt
    Skills              []PromptSkill   // loadable skills for the prompt
    HasSkills           bool            // true when read tool is active AND skills exist
    Rules               []PromptRule    // domain rules with glob patterns
    HasRules            bool            // true when domain rules exist
    GenericRules        []string        // unconditional sticky rule bodies
    HasGenericRules     bool            // true when generic rules exist
    Guidelines          []string        // session guidelines rendered as bullet points
    Personality         string          // optional <personality> block content
    HasRead             bool            // true when "read" is in the active tool set
    HasBash             bool            // true when "bash" is active
    HasEdit             bool            // true when "edit" is active
    HasWrite            bool            // true when "write" is active
    HasGrep             bool            // true when "grep" is active
    HasFind             bool            // true when "find" is active
    HasLs               bool            // true when "ls" is active
    HasTodoWrite        bool            // true when "todo_write" is active
    HasTask             bool            // true when "task" is active
    HasSkill            bool            // true when "skill" is active
    HasWebSearch        bool            // true when "web_search" is active
    HasWebFetch         bool            // true when "web_fetch" is active
    HasAttachment       bool            // true when "attachment" is active
}
```

`HasCustomPrompt` gates the entire template body. When true, the default role,
principles, runtime, execution workflow, and delivery contract are replaced by
`CustomPrompt`; only capabilities, appended content, context, skills, and the
date/cwd footer survive.

`CapabilitiesSection` is pre-rendered — the template injects it as a raw string
(`{{.CapabilitiesSection}}`). This keeps the `<available_tools>` block
authoritative and self-contained rather than scattering its logic across template
conditionals.

`HasSkills` has a double gate: the `read` tool must be active *and* at least one
skill must exist. This prevents advertising `skill://` reads when the model
cannot perform them.

### BuildSystemPrompt (`internal/prompt/prompt.go:39-86`)

Assembles a `PromptData` from the caller's options and renders the template.

```go
func BuildSystemPrompt(options BuildSystemPromptOptions) string
```

**`options ProductName`** — defaults to `"Harness"` when empty
(`defaultProductName`, `internal/prompt/prompt.go:88-93`).

**`options CustomPrompt`** — when non-empty, `HasCustomPrompt` becomes true and
the template replaces the entire default body with this value.

**`options SelectedTools`** — the ordered list of active tool names. Copied
(defensively) before use; drives `CapabilitiesSection`, every `HasXxx` boolean,
and guideline generation.

**`options ToolSnippets`** — a map from tool name to one-line description.
Feeds `CapabilitiesSection`; a tool whose snippet is missing is silently dropped
from the `<available_tools>` block. The production path (`BuildToolSnippets` in
`internal/session/session.go:54-89`) always supplies a snippet for every active
tool, so this is a defense-in-depth measure, not a production code path.

**`options Cwd`** — converted to forward slashes with `filepath.ToSlash` before
rendering.

**`options Now`** — controls the date footer. When zero, defaults to
`time.Now()`. Injected so tests can freeze the date.

**`options Skills`** — filtered through `visiblePromptSkills` which excludes
skills with an empty name or description.

**Returns** the fully rendered system prompt as a single string. The template
execution cannot fail at runtime — `template.Must` at init time guarantees valid
syntax, and the typed struct guarantees every referenced field exists.

### RebuildSystemPrompt (`internal/session/session.go:118-139`)

The session-level entry point. Derives tool names and snippets from the live
`agent.AgentTool` slice and delegates to `prompt.BuildSystemPrompt`.

```go
func RebuildSystemPrompt(opts RebuildSystemPromptOptions) string
```

**`opts ActiveTools`** — the live `[]agent.AgentTool` slice from the tool
registry. Names are extracted via `ToolNames`; snippets via `BuildToolSnippets`
(unless the caller supplies its own). This is the single source of truth: the
prompt can only name tools that exist in this slice.

**`opts ToolSnippets`** — when nil, `BuildToolSnippets` generates descriptions
from the tool's `Name` field with hardcoded descriptive labels for known tools.
The override exists for callers with non-standard tool names.

### CapabilitiesSection (`internal/prompt/prompt.go:114-141`)

Renders the authoritative `<available_tools>` block and the `<runtime>` notes
block.

```go
func CapabilitiesSection(names []string, snippets map[string]string) string
```

**`names`** — ordered tool names. Order is preserved in output; the test
`TestBuildSystemPromptToolOrderPreserved` asserts this.

**`snippets`** — one-line descriptions keyed by tool name. A nil map or a
missing entry silently skips that tool from the listing. The production path
always provides snippets, so this guards against a caller constructing a partial
map by accident.

**Returns** a string containing:

1. `<available_tools>` — one `- name: description` line per tool, preceded by a
   preamble instructing the model to use exactly these tools and not invent
   unavailable ones.
2. `<runtime>` — sandbox ephemerality, network caveats, and (when `read` is
   active) a note about truncated tool output with the path to the full result.
   When `todo_write` is active, mentions `.harness/progress.md`.

Returns `""` when no tools have snippets, which means the rendered prompt will
have no `<available_tools>` block at all.

### Supporting types

```go
type PromptTool struct {
    Name  string
    Label string
}

type PromptSkill struct {
    Name        string
    Description string
    Location    string
}

type PromptRule struct {
    Name        string
    Description string
    Globs       []string
}

type ContextFile struct {
    Path    string
    Content string
}
```

`PromptSkill.Location` is a `skill://` URI (`visiblePromptSkills` at
`internal/prompt/prompt.go:95-107` converts the disk path to a `skill://`
scheme), never a filesystem path. This keeps the model-facing prompt portable
and prevents leaking host filesystem details.

## How it works

The data flow from stack construction to rendered prompt has four layers.

### Layer 1: stack assembly picks active tools

`BuildAgentStack` (`internal/session/stack.go:71-162`) builds the full tool
registry with `tools.AllTools`, then filters it through `SelectActive`. Plan
mode gets a narrow allowlist (`read`, `grep`, `find`, `ls`, `web_search`,
`web_fetch`, `attachment`); normal mode gets everything minus exclusions. The
resulting `active []agent.AgentTool` slice is passed to `RebuildSystemPrompt`.

### Layer 2: session derives names and snippets

`RebuildSystemPrompt` (`internal/session/session.go:118-139`) calls
`ToolNames(active)` to extract the ordered name list and
`BuildToolSnippets(active)` to build the description map. It then constructs a
`prompt.BuildSystemPromptOptions` and delegates to `BuildSystemPrompt`.

`BuildToolSnippets` (`internal/session/session.go:54-89`) has hardcoded
descriptions for known tools (`"read" → "Read file contents"`, `"bash" →
"Execute bash commands (ls, grep, find, etc.)"`, etc.) and falls back to the
tool's `Label` or `Name` for unknown tools. This ensures every active tool gets
a snippet even if the tool definition's label field is empty.

### Layer 3: prompt assembles typed data

`BuildSystemPrompt` (`internal/prompt/prompt.go:39-86`) does four things:

1. Copies `SelectedTools` defensively.
2. Builds `PromptTool` values (name + label/snippet) for every tool.
3. Populates `PromptData` — computes every `HasXxx` boolean from the tool list
   with `hasTool`, renders `CapabilitiesSection` from the names and snippets,
   filters skills through `visiblePromptSkills`, compacts generic rules, and
   builds guidelines via `buildGuidelines`.
4. Calls `renderSystemPrompt(data)`.

### Layer 4: template renders

`renderSystemPrompt` (`internal/prompt/system.go:79-85`) executes the
`text/template` against a `strings.Builder`. The template is the final authority
on structure and ordering; the data struct only controls what appears.

The template's conditional blocks work as follows:

- `{{if .HasCustomPrompt}}` — short-circuit: emit `CustomPrompt` and skip the
  default role/principles/runtime/execution/delivery body. Personality is also
  skipped. Capabilities, append content, context, skills, date, and cwd are
  always emitted regardless.

- Each `HasXxx` boolean gates a tool-specific instruction in the TOOL POLICY
  section. For example, `{{if .HasRead}}` emits "File or directory reads: use
  read." If `read` is not in the active set, that line does not appear and the
  model is not told to use a tool it does not have.

- `{{range .Skills}}` iterates the skill list, emitting one `<skill>` element
  per skill with XML-escaped name, description, and location.

- `{{range .Guidelines}}` emits bullet points under `# Session Guidelines`.
  `buildGuidelines` always appends "Be concise in your responses" and "Show file
  paths clearly when working with files" plus a bash-fallback guideline when
  bash is active but grep/find/ls are not.

## Failure modes and invariants

### Invariant: the prompt never names an unavailable tool

Enforced by construction. The tool names in `<available_tools>` come from
`ToolNames(active)`, which reads `tool.Name` from each `agent.AgentTool` in the
active set. Every `HasXxx` boolean is computed by `hasTool` against the same
tool name list. There is no hand-written prose tool list.

The test `TestBuildSystemPromptProjectsOnlyActiveCapabilities` builds a prompt
with only `read` and `grep` and asserts that `bash`, `write`, and unavailable
schemes (`vault://`, `mcp://`, `xd://`) are absent from the output.

### Invariant: skills only appear when the read tool is active

`HasSkills` requires both `hasTool(tools, "read")` and `len(promptSkills) > 0`.
If the session has no read tool, the `<available_skills>` block is absent even
when skills are configured. This prevents the model from being told about
`skill://` resources it cannot load.

`TestBuildSystemPromptSkillsOnlyWithRead` builds two prompts — one with `read`
active and one without — and asserts skills appear only in the first.

### Invariant: tool order is preserved

`CapabilitiesSection` iterates `names` in the order supplied. The caller
(`ToolNames`) preserves the order of `active []agent.AgentTool`.
`TestBuildSystemPromptToolOrderPreserved` passes `["bash", "read"]` and asserts
`bash` appears before `read` in the output.

### Invariant: custom prompt retains capabilities

When `HasCustomPrompt` is true, the template replaces the role/principles/runtime
body but still injects `CapabilitiesSection`, `AppendSystemPrompt`,
`ContextFiles`, skills, date, and cwd. The `TestBuildSystemPromptCustomReplacesTemplate`
test verifies that custom text appears and default blocks are absent, while
`TestBuildSystemPromptCustomIncludesCapabilities` verifies that
`<available_tools>` and `<runtime>` appear alongside custom text.

### Failure mode: template parse error

The template is parsed with `template.Must` at package initialization. A syntax
error in `system.md.tmpl` causes a `panic` at import time — the binary will not
start. This is intentional: a malformed template is a compile-time error, not a
runtime surprise.

### Failure mode: template execution error

`renderSystemPrompt` panics on execution failure with a message identifying the
embedded template. In practice, execution cannot fail because the typed struct
guarantees every referenced field exists and has the correct type. A panic here
indicates a programming error (e.g., adding a template field without adding the
corresponding struct field), which `go build` would catch.

### Failure mode: XML injection

All user-supplied values rendered inside XML tags — paths, skill names,
descriptions, context file content — pass through the `xml` template function,
which escapes `&`, `<`, `>`, `"`, and `'`. A path containing `<script>` becomes
`&lt;script&gt;` in the rendered prompt.

### Failure mode: missing snippet

`CapabilitiesSection` skips a tool when its snippet is empty or nil. In
production, `BuildToolSnippets` supplies a snippet for every active tool (with a
fallback to the tool's name), so this path is not exercised. The review at
`phase2-review.md:85` flagged this as a nit: the typed API can render an
incomplete tool list if given a partial snippet map. The fix (noted as deferred)
would make the section total — include every tool with a safe fallback rather
than silently dropping.

### Failure mode: guideline deduplication

`buildGuidelines` uses a `seen` map to deduplicate guidelines. The same
guideline added via `PromptGuidelines` and the built-in defaults produces only
one entry. This prevents token waste from repeated boilerplate.

## TypeScript to Go

A TypeScript reference agent would assemble a system prompt through template
literals and conditional string concatenation:

```typescript
// TypeScript instinct: build the prompt by string concatenation
function buildSystemPrompt(tools: Tool[]): string {
  let prompt = `You are a helpful assistant.\n\n`;
  prompt += `<available_tools>\n`;
  for (const tool of tools) {
    prompt += `- ${tool.name}: ${tool.description}\n`;
  }
  prompt += `</available_tools>\n`;

  if (tools.some(t => t.name === "read")) {
    prompt += `Use the read tool for file operations.\n`;
  }
  if (tools.some(t => t.name === "bash")) {
    prompt += `Use bash for shell commands.\n`;
  }
  // ... more conditionals scattered across the function

  return prompt;
}
```

This works but has no structural guard. If a tool conditional is added to the
prompt string but the corresponding tool check is wrong (`t.name === "reed"`),
the bug is invisible until someone inspects a rendered prompt. If the tool list
and the prompt prose are maintained in different files, drift is inevitable.

Go forces a different approach:

1. **`go:embed` replaces runtime file I/O.** The template is compiled into the
   binary. No filesystem reads at prompt-build time, no deployment path issues,
   no risk of a missing template file in production.

2. **`text/template` replaces string concatenation.** The template is a single
   Markdown file with `{{if}}` and `{{range}}` directives. The structure is
   visible in one place; adding a conditional means adding exactly one
   `{{if .HasXxx}}` block, not tracking down every string concatenation site.

3. **A typed struct replaces ad-hoc booleans.** In TypeScript, you might
   compute `hasRead = tools.some(t => t.name === "read")` inline and hope the
   check stays in sync with the string. In Go, `HasRead` is a struct field. If
   the template references `.HasRead` but the struct omits it, `go build` fails.
   If the struct sets `HasRead` but the template forgets the `{{if}}`, a test
   catches it. The contract is bilateral and compiler-enforced.

4. **`CapabilitiesSection` is a pure function of the tool list.** No
   hand-written tool names in prose. The `<available_tools>` block is a
   mechanical rendering of `names []string` — the same names that came from the
   registry. TypeScript could do the same, but the language's dynamic nature
   makes it easy to bypass; Go's lack of dynamism makes it the only path.

5. **The `xml` template function is a `strings.NewReplacer`, not a regex.** It
   replaces exactly five characters with their XML entities. No regex
   compilation, no allocation per replacement beyond the output string builder.
   The replacer is a package-level `var` initialized once.

The result: a prompt builder where the distance between "the registry has a tool
named X" and "the prompt mentions tool X" is one function call
(`hasTool(tools, "X")`) and one struct field assignment (`HasX: true`). The
compiler and the test suite enforce the connection at every level.
